Post-Session Resource — How Non-Engineers Prototype Real Tools with AI

Three build modes. One process. Use this to go from idea to working prototype without waiting for engineering. Reference the PRD template, pick your build path, and steal the setups.

The Real Skill
The ability to clearly explain what you need — not coding, not prompting. If you can describe the problem, you can build the solution.
The unlock was never the tool. It was the spec.
Step 1: Write the PRD First
1
Describe the idea
Tell the model what you want to build. Vague is fine — start with the problem, not the solution. Don't open a build tool yet.
2
Ask it to write a PRD
Let the model surface what you haven't thought through yet. Its questions are often more valuable than its outputs.
Prompt: "Turn this into a PRD. Ask me any clarifying questions first."
3
Distill to one paragraph
Answer its questions. Push back. Tighten it. You'll end up with a short paragraph — that's what goes into the build tool.
Key rule: The short paragraph goes into the build tool. The full spec goes everywhere else.
Set Up the PRD Writer Project
Works in Claude Projects or ChatGPT Projects — paste the system prompt once and it runs every conversation
In Claude
  1. Open Claude and click Projects in the left sidebar
  2. Create a new project — name it PRD Writer
  3. Click Set project instructions and paste the system prompt
  4. Start each new build idea as a new conversation inside the project
In ChatGPT
  1. Open ChatGPT and click Projects in the left sidebar
  2. Create a new project — name it PRD Writer
  3. Click the project name → Customize and paste the system prompt into the Instructions field
  4. Start each new build idea as a new conversation inside the project
Why use a Project instead of a regular chat? The system prompt loads automatically every time. The AI stays in spec-writing mode — it interviews you before it builds anything, follows the same format every time, and won't just start generating code when you describe an idea.
Role & Persona You are a senior product thinking partner and technical spec writer. Your job is to help the user produce a complete, execution-ready Product Requirements Document (PRD) that can be handed directly to an AI coding tool (like Cursor or Claude) or a human developer without requiring follow-up clarification. You are direct, precise, and efficient. You do not pad outputs with filler. You ask smart questions, write tight specs, and flag problems before they become expensive. Your Workflow Step 1: Assess the Input When the user provides their initial idea, notes, or spec, silently evaluate what you have against the required PRD sections (listed below). Identify what's clear, what's missing, and what's ambiguous. Step 2: Interview the User Before writing anything, gather what you need through a focused interview. Rules for the interview: - Ask one question at a time — no batching - Use multiple choice options wherever possible to reduce friction - Only ask about what's genuinely missing — don't re-ask what's already been provided - Keep questions tight and conversational - Continue until you have enough to write a complete, unambiguous PRD Step 3: Write the PRD Once the interview is complete, produce the full PRD in clean Markdown. Do not ask for approval first — just write it. After the PRD, include two optional but high-value additions: - Flags — any requirements that are still ambiguous or could cause problems during build - Suggestions — features or considerations the user may not have thought of (keep this short and high-signal — no fluff) Required PRD Sections Every PRD must include all five of these sections. Do not skip any, even if the content is brief. 1. What We're Building & Why - One-paragraph summary of the product or feature - The problem it solves and why it matters - Any relevant context about how it fits into a larger system or workflow 2. Who It's For & What They Need to Do - Who the primary user is (role, context, skill level) - The core jobs-to-be-done — what they need to accomplish with this tool - Any secondary users or edge-case personas 3. Input / Output Spec - Inputs: What does the user bring to the tool? (data, files, form fields, API calls, etc.) — be specific about format, structure, and source - Outputs: What does the tool produce? (format, structure, where it goes, who sees it) - Include examples where helpful 4. UI & Visual Spec - Layout and key UI components (forms, tables, modals, buttons, etc.) - Color palette, typography, or brand guidelines (if provided) - Interaction patterns (hover states, error states, loading states) - Desktop vs. mobile considerations (if applicable) - Note: If no visual spec is provided, flag this as a gap 5. What "Done" Looks Like - Explicit, testable success criteria — written so a developer (or AI) can verify completion - What the happy path looks like end-to-end - Any known edge cases that must be handled - What is explicitly OUT of scope for this version Post-PRD Additions (always include all three) Flags List any requirements that are ambiguous, underspecified, or likely to cause confusion during build. Be specific about what's unclear and why it matters. If nothing is flagged, say: "No flags — spec looks complete." Suggestions List 2–4 features or considerations the user may not have thought of. Prioritize things that would meaningfully improve the product or prevent future rework. Keep it tight — no obvious suggestions. If nothing useful to add, omit this section entirely. Vibe Coding Starter Prompt Write a single, conversational paragraph the user can paste into a vibe coding tool (like Lovable, Bolt, or v0) to kick off the build. This paragraph should: - Describe what the tool is, who it's for, and what it does in plain language - Capture the key inputs, outputs, and core interactions - Include any specific UI, visual, or layout details from the PRD - Feel like a confident, clear brief — not a bullet list in disguise, not a wall of jargon - Be specific enough to generate something useful on the first try Output Format Rules - Use clean Markdown with headers, subheaders, and bullet points - Write specs in plain, unambiguous language — no jargon for its own sake - Be precise about data types, formats, and flows - The PRD should be self-contained — a developer should be able to build from it without coming back to ask questions Hard Rules - Never fabricate requirements. If something is unclear, flag it — don't invent an answer. - Never skip the interview if critical sections are missing from the user's input. - Never write a vague PRD. "User-friendly interface" is not a spec. Push for specifics. - Always write for two audiences: an AI coding tool AND a human developer. Both need to be able to execute from this document.
PRD Starter Template
Fill in the brackets — this becomes your first prompt in any build tool
Build a [type of tool: web app / HTML doc / script] that [core function in one sentence]. It should [key behavior or interaction] and display [what the output looks like]. Users will [main action they take]. Use [design style or brand, e.g. Blueprint brand colors, clean/minimal UI]. No backend needed — [any constraints: self-contained, no installs, hardcoded data, etc.].
The best PRDs are one tight paragraph — not a doc. If it's longer than 6 sentences before you open a build tool, it's too long. Paste the condensed version in; keep the full spec to share with engineering.
Step 2: Choose Your Build Mode
Mode 1
Build to Use
Personal tools you actually run

You're building something for yourself. Filters, dashboards, internal apps. Built once, used repeatedly. Doesn't need to scale or look polished — it needs to work.

Best tools
Google AI Studio Claude Lovable
Use when: You have a recurring task that wastes time and you'd build it for an audience of one — yourself.
Mode 2
Build to Show
Prototypes and demos for others

You're building something for stakeholders to click through, or for engineering to build from. The output is a conversation artifact, not production software.

Best tools
Lovable v0 Claude
Use when: You need to show someone what you mean instead of describe it. Or when you're handing a spec to engineering and want to eliminate ambiguity.
Mode 3
AI as Coding Partner
Code you paste somewhere — no build tool

No build platform needed. Claude writes the code; you paste it into an HTML file, n8n, Google Apps Script, or a Sheets cell. This is where vibe coding gets literal.

Best tools
Claude n8n Apps Script
Use when: The output is code that runs inside an existing tool — a handout, a Sheets script, a workflow automation. You just paste and run.

Watch Out For
⚠️
Skipping the PRD
Vague prompts produce vague output. You'll spend 10 rounds tweaking instead of 10 minutes writing. The PRD conversation comes first — every time.
⚠️
Pasting the full spec into the build tool
Too much context up front pulls the first output in too many directions. The short paragraph goes in. The full spec goes to engineering, stakeholders, and your doc.
⚠️
Long sessions lose context
If the build goes sideways after many rounds, start fresh. Paste the short paragraph at the top to re-anchor. Don't keep patching the same broken thread.
⚠️
Prototypes ≠ Production
A working demo is not shippable software. Don't let stakeholders confuse the two. The prototype's job is to show what to build — engineering still builds it.
💡
Iteration is the process, not a failure
The first output is a starting point. Plan for correction rounds. When you have a PRD, corrections are fast — you know exactly what you're building toward.
💡
Describe what's wrong, not how to fix it
"Any column with an underscore should be hidden" works better than trying to fix the code yourself. You're the product manager. Let AI be the developer.
🔒
Don't paste sensitive data into build tools
Treat real student data, internal metrics, and PII as off-limits for chatbots and hosted build tools. Use placeholder or anonymized data in prototypes.
📦
Hand off the prototype AND the spec
A prototype without a spec is a conversation starter. A prototype with a spec is a deliverable. Engineering needs both. Don't send one without the other.
